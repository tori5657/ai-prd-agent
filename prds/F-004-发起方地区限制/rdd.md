# RDD：发起方地区限制

## 用户故事（已确认）

As a 工作空间超级管理员，I want to 预设禁止发起信封的国家/地区列表，当发起方从受限地区（如俄罗斯）访问时，系统自动拦截其发起操作，So that 规避数据跨境合规风险，防止合同数据因在受限国家发起而触犯当地数据出入境法规。

---

## 节 1：需求摘要

| 维度 | 内容 |
| --- | --- |
| 业务对象 | 信封发起行为；工作空间级地区限制配置 |
| 用户角色 | 核心：工作空间超级管理员（配置者）；受影响：发起方（Sender） |
| 使用场景 | 客户在俄罗斯等有严格数据出入境法规的国家开展业务，需防止员工从境内发起信封，避免合同数据因跨境流动违反当地法规 |
| 核心动作 | 管理员：在 Settings > Preference 配置禁止国家/地区列表；系统：发起方进入发起流程时及点击发送时双重校验 IP，命中禁止列表则拦截 |
| 预期价值 | 客户无需依赖内部 IT 管控，由平台层保障发起侧数据跨境合规；与现有签署侧地区限制形成完整的双端合规覆盖 |
| 成功指标 | 合规场景下，受限地区的发起方 100% 被拦截且无法通过前端或 API 绕过；管理员可在 Preference 完成配置，无需工单支持 |
| 需求优先级 | P1 |

---

## 节 2：初步方案

### 方案边界

- **本版做**：在现有预备设置中的「签署人国家/地区限制」配置项上扩展，配置生效后同时控制签署侧（签署人访问/提交）和发起侧（发起人进入发起流程/点击发送）；双节点 IP 校验
- **本版不做**：新增独立配置项；API 发起限制；VPN 精确识别；拦截事件审计日志
- **后续规划**：审计日志按需补充

### 功能模块

| 模块名 | 类型 | 说明 |
| --- | --- | --- |
| 预备设置 — 签署人国家/地区限制配置项 | 调整 | 扩展现有配置项的控制范围，配置后同时生效于发起侧和签署侧；名称/描述文案需相应更新 |
| 发起流程入口 IP 校验 | 新增 | 发起方进入发起页时校验 IP，命中限制列表则整页拦截 |
| 发送节点 IP 校验 | 新增 | 发起方点击发送时再次校验 IP，命中则 Toast 提示并阻止发送，已填内容保留 |

### 页面结构

```
预备设置（Pre-made Settings）
└── 签署人国家/地区限制（调整现有配置项）
    ├── 名称/描述文案更新（体现同时控制发起侧）
    └── 其余配置项结构不变（国家/地区多选列表）

信封发起流程
├── 进入发起页 → IP 校验失败 → 整页拦截（参考签署侧访问拦截样式）
└── 点击发送 → IP 校验失败 → Toast 提示，阻止发送，已填内容保留
```

### 核心流程

- **配置流程**：超级管理员 → 预备设置 → 签署人国家/地区限制 → 配置允许/禁止国家 → 保存（已有流程不变）
- **发起拦截流程**：发起方进入发起页 → 系统校验 IP → 命中限制 → 整页拦截，展示提示文案
- **发送拦截流程**：发起方填完配置点击发送 → 系统再次校验 IP → 命中限制 → Toast 提示，阻止发送
- **正常流程**：IP 未命中限制，或配置列表为空 → 正常进入/发送

### 关键设计决策

| 决策点 | 方案 A | 方案 B | 推荐 & 理由 |
| --- | --- | --- | --- |
| 配置项复用 vs 新建 | 复用现有「签署人国家/地区限制」，扩展其控制范围 | 新增独立「发起方地区限制」配置项 | **推荐 A**：同一套国家列表同时约束发起和签署，避免管理员维护两份配置；语义上"来自受限地区的人既不能发起也不能签署"更一致 |
| 双节点校验 | 进入时 + 发送时均校验 | 仅发送时校验 | **推荐 A**：进入时校验避免用户填完表单才被拦截；发送时校验防止进入后切换 IP 绕过 |
| API 是否受限 | API 同样校验 | API 豁免 | **推荐 B（本期）**：API 场景复杂度高，本期不纳入范围 |

### 风险提示

- P1：VPN / 数据漫游影响 IP 定位准确性，可能导致误判——与签署侧同样局限，在配置项说明中注明，由客户知情承担
- P2：现有配置项文案需同步更新，避免用户误以为配置仅影响签署侧；需评估对存量已配置客户的理解影响
- P3：配置列表为空时，需在 UI 中明确提示"当前无限制，所有地区均可发起和签署"，避免管理员误解
