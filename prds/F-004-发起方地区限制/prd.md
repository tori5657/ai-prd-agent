## 3 需求概要

### 3.1 问题与机会（概要）

俄罗斯等国家对数据出入境有严格限制，若发起方从这些国家创建并发送信封，合同数据可能触犯当地数据本地化法规。现有「签署人国家/地区限制」功能已能控制签署侧的地区合规，但尚未覆盖发起侧。本需求在「偏好设置 > 签署 Tab」中新增独立配置项「发起方国家/地区限制」，专门控制发起侧的地区合规，与现有签署侧配置相互独立，避免配置项冲突及对历史客户的意外影响。

### 3.2 目标用户（概要）

- 核心用户：工作空间超级管理员（配置者）
- 受影响用户：发起方 Sender（在发起流程中受 IP 校验约束）

### 3.3 方案概述

在「设置 > 偏好设置 > 签署 Tab」中新增独立配置项「发起方国家/地区限制」，与现有「签署人国家/地区限制」并列，互不干扰。配置生效后，系统在发起方点击各发起入口时和点击发送时各执行一次 IP 地区校验，命中限制均以 Toast 形式提示并阻止操作，用户停留在当前页面。覆盖的发起入口包括：单方签、单个发起、批量发起、草稿发起、信封编辑、信封模版发起，以及 Lark 多维表发起（P2）。

### 3.4 成功指标（3-5 项）

| 指标 | 目标值 | 观测时间 | 数据来源 |
| --- | --- | --- | --- |
| 受限地区发起方被拦截率 | 100%（进入节点 + 发送节点均拦截） | 上线后首个完整周 | 服务端 IP 校验日志 |
| 管理员无需工单完成配置 | 新配置项描述清晰，0 工单咨询 | 上线后首月 | 客服工单系统 |
| 现有签署侧拦截行为无回归 | 签署侧拦截通过率 = 上线前基线 | 上线后首周 | 签署端 IP 校验日志 |

## 6 用户故事与用例

### 6.1 Epic

在「偏好设置 > 签署 Tab」新增独立配置项，使管理员能够单独控制发起侧的地区合规，与签署侧配置互不干扰，满足数据跨境合规要求。

### 6.2 Must Have（MVP）

**故事 1：管理员新建发起方地区限制配置**

```text
作为工作空间超级管理员，
我希望在偏好设置中看到独立的「发起方国家/地区限制」配置项，
支持一键全选和取消全选，
以便我单独管理发起侧的地区合规，不影响现有签署侧配置。
```

验收标准：
- Given 管理员进入「设置 > 偏好设置 > 签署 Tab」
- When 查看配置项列表
- Then 看到「发起方国家/地区限制」独立配置项，与「签署人国家/地区限制」并列展示
- And 国家/地区多选下拉支持一键全选与一键取消全选操作

**故事 2：发起方在入口点击时被地区拦截**

```text
作为从受限地区尝试发起信封的发起方，
我希望在点击发起入口时立即收到 Toast 提示，
并停留在当前页面，
以便我知道当前操作不被允许，以及应如何处理（检查网络 / 联系管理员）。
```

验收标准：
- Given 工作空间「发起方国家/地区限制」已配置非空列表，且发起方当前 IP 在限制列表中
- When 发起方点击任意发起入口（单个发起、批量发起、单方签、草稿发起、信封编辑、信封模版发起）
- Then 显示 Toast 错误提示，用户停留在当前页面，不进入发起流程

- Given 工作空间「发起方国家/地区限制」列表为空
- When 发起方点击任意发起入口
- Then 正常进入发起流程，无拦截

**故事 3：发起方在发送节点被地区拦截**

```text
作为已进入发起流程但在点击发送前切换至受限网络的发起方，
我希望在点击发送时收到 Toast 提示并保留已填写内容，
以便我了解发送失败原因，并在切换网络后重试，不丢失填写进度。
```

验收标准：
- Given 发起方已进入发起流程完成配置，但当前 IP 不在工作空间允许列表中
- When 发起方点击发送按钮
- Then 显示 Toast 错误提示，发送操作被阻止
- And 已填写的信封配置内容完整保留，不丢失

- Given 发起方 IP 在允许列表中
- When 发起方点击发送按钮
- Then 正常发送信封

### GEO-CFG-001 偏好设置-签署Tab-新建发起方国家地区限制配置项 需求说明

#### 8.1 任务故事（Job Story）

当管理员在「设置 > 偏好设置 > 签署 Tab」中配置发起侧合规策略时，我想要看到独立的「发起方国家/地区限制」配置项（与现有「签署人国家/地区限制」并列），且国家/地区多选支持一键全选与一键取消全选，这样我可以独立控制发起侧地区限制，不影响签署侧已有配置。

#### 8.2 逻辑实现规范

##### Context（前置条件）

- 管理员已进入「设置 > 偏好设置 > 签署 Tab」
- 「发起方国家/地区限制」配置项为新增条目，初始为空（不继承签署侧配置值）

##### Action（触发动作）

1. 管理员查看新配置项名称「发起方国家/地区限制」及描述
2. （可选）管理员展开国家/地区多选下拉
3. （可选）管理员点击「全选」——选中所有国家/地区
4. （可选）管理员点击「取消全选」——清空所有已选国家/地区
5. （可选）管理员搜索特定国家/地区后进行单选
6. 管理员保存配置

##### Outcome（预期结果）

1. **界面变化**：「发起方国家/地区限制」配置项出现在「签署人国家/地区限制」附近（同一 Tab），两者并列独立；描述文案说明该配置仅控制发起侧地区合规；多选下拉列表顶部展示「全选」与「取消全选」操作项
2. **全选行为**：点击「全选」后所有国家/地区均被勾选；若已有部分选中，全选补全剩余项
3. **取消全选行为**：点击「取消全选」后所有已选项清空，配置回到空值状态（等同于限制未启用）
4. **数据变化**：新增独立配置存储字段，与签署侧配置完全隔离；历史客户「签署人国家/地区限制」配置值不受影响
5. **反馈提示**：保存成功后展示现有 Toast 反馈

#### 8.3 异常处理要求

| 异常场景 | 触发条件 | 系统行为（预期结果） |
| --- | --- | --- |
| 存量客户上线后 | 上线前已配置「签署人国家/地区限制」的工作空间 | 签署侧配置值完全不变；「发起方国家/地区限制」初始为空，发起侧无限制，不影响任何已有发起行为 |
| 全选后保存 | 管理员点击全选并保存 | 所有国家/地区均被限制，任何地区的发起方均无法发起 |
| 取消全选后保存 | 管理员点击取消全选并保存 | 配置为空，发起侧限制不生效，所有地区均可正常发起 |

### GEO-INIT-001 ～ GEO-INIT-007 发起流程-各入口-发起方IP地区校验拦截 需求说明

> GEO-INIT-001～007 逻辑完全一致，仅触发入口不同，统一描述如下。

#### 覆盖入口一览

| 功能编号 | 入口名称 | 入口描述 | 优先级 |
| --- | --- | --- | --- |
| GEO-INIT-001 | 单个发起 | 新建单个信封，进入发起配置页 | P1 |
| GEO-INIT-002 | 批量发起 | 进入批量发起流程入口页 | P1 |
| GEO-INIT-003 | 单方签 | 进入单方签（自签）流程入口页 | P1 |
| GEO-INIT-004 | 草稿发起 | 从草稿列表点击「继续发起」进入草稿编辑页 | P1 |
| GEO-INIT-005 | 信封编辑 | 进入已发出信封的编辑页；确认/保存时再次校验（双节点） | P1 |
| GEO-INIT-006 | 信封模版发起 | 从信封模版点击「使用模版发起」进入配置页 | P1 |
| GEO-INIT-007 | Lark 多维表发起 | 从 Lark 多维表插件触发信封发起 | P2 |

#### 8.1 任务故事（Job Story）

当发起方通过以上任意入口进入信封发起流程时，我想要系统立即校验其 IP 地区，若不在允许列表则展示 Toast 错误提示并阻止进入，这样我可以在发起方填写任何内容之前阻止受限操作，避免用户填完表单才被告知无法提交。

#### 8.2 逻辑实现规范

##### Context（前置条件）

- 工作空间「发起方国家/地区限制」配置列表非空
- 发起方点击上表中任意一个入口

##### Action（触发动作）

1. 发起方点击入口，在跳转前触发 IP 校验，系统获取发起方当前 IP
2. 系统将 IP 映射到地理位置
3. 系统比对该地理位置与工作空间「发起方国家/地区限制」配置列表（黑名单）

##### Outcome（预期结果）

1. **IP 不在限制列表**：正常跳转至对应发起页，无额外提示
2. **IP 在限制列表**：展示 Toast 错误提示，阻止跳转；用户停留在当前页面
3. **列表为空**：限制未启用，正常跳转

#### 8.3 异常处理要求

| 异常场景 | 触发条件 | 系统行为（预期结果） |
| --- | --- | --- |
| IP 地理位置服务不可用 | IP 映射服务超时或报错 | 降级拦截：IP 无法识别时按受限处理，展示 Toast 提示阻止操作（安全优先） |
| 配置列表为空 | 管理员未配置任何国家/地区 | 视为限制未启用，正常放行 |
| IP 受 VPN 影响 | 发起方通过 VPN 访问，实际 IP 与物理地区不符 | 以系统检测到的 IP 为准，不做额外处理；配置项文案已注明此局限性 |

### GEO-SEND-001 发起流程-发送节点-发起方IP地区校验拦截 需求说明

#### 8.1 任务故事（Job Story）

当发起方完成信封配置并点击发送时，我想要系统再次校验其 IP 地区，这样我可以防止发起方进入流程后切换至受限网络再完成发送。

#### 8.2 逻辑实现规范

##### Context（前置条件）

- 发起方已通过入口 IP 校验，正常进入发起流程并完成配置
- 发起方点击发送按钮

##### Action（触发动作）

1. 发起方点击发送，系统重新获取当前 IP
2. 系统重新执行地区校验（同 GEO-INIT-001 逻辑）
3. 返回校验结果

##### Outcome（预期结果）

1. **IP 在允许列表**：正常发送信封

## 9 非功能性需求

### 9.1 性能要求

| 场景 | 要求 |
| --- | --- |
| IP 地区校验响应时间 | 不增加发起页加载时间超过 200ms（P95） |
| 发送节点校验响应时间 | Toast 提示在点击发送后 1s 内展示 |

### 9.2 安全要求

- IP 校验在服务端执行，不依赖客户端上报
- 不在前端日志或 URL 中暴露允许地区列表

### 9.3 可用性与可访问性

- 拦截提示文案须包含处理建议（检查网络 / 联系管理员），不得仅展示"无权操作"
- Toast 提示需满足 WCAG 对比度要求

### 9.4 兼容性要求

本功能 IP 校验在服务端执行，客户端无差异，全端行为一致。

### 9.5 数据统计需求

| 事件对象 | 触发动作 | 事件名 | 属性 | 属性值说明 |
| --- | --- | --- | --- | --- |
| 发起入口 | 点击入口触发 IP 校验失败，Toast 展示 | geo_block_init_toast | workspace_id, detected_region, entry_point | 用于统计各入口拦截频次；entry_point 枚举：single/batch/self_sign/draft/edit/template/lark |
| 发起页 | 发送节点 IP 校验失败，Toast 展示 | geo_block_send_toast | workspace_id, detected_region | 用于统计发送节点拦截频次 |
