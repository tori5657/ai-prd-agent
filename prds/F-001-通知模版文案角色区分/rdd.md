# RDD：通知模版文案角色区分

## 用户故事（已确认）

As a **文档流程中的审批人**，
I want to **收到标明"待处理"而非"待签署"的通知**，
So that **我能准确判断自己需要做什么操作，不再因通知文案与实际任务不符而产生误解**。

---

## 节 1：需求摘要

| 维度 | 内容 |
| --- | --- |
| 业务对象 | 通知消息模版（短信/邮件/Lark/WhatsApp/站内信） |
| 用户角色 | 核心：信封签署方（含审批人）/ 次级：信封发起方、信封抄送方 |
| 使用场景 | 系统在签署任务推送、过期提醒、催签、完成、取消、拒签等事件时向相关方发送通知 |
| 核心动作 | 将通知模版中"签署"等特指签字操作的词替换为"处理/操作/完成"等通用词，以适配审批人等非签名角色 |
| 预期价值 | 消除审批人看到"待签署"时产生的角色认知混乱，降低因误解通知内容导致的操作失误和客诉 |
| 成功指标 | 上线后所有涉及模版条目按 Excel《消息通知模板.xlsx》"本次调整标注" tab 完成替换，无漏改/误改 |
| 需求优先级 | P1 |

---

## 节 2：初步方案

### 方案边界

- **本版做**：按《消息通知模板.xlsx》"本次调整标注" tab 中标记的 309 条文案，完成 5 个渠道、17 种语言的逐条文案替换
- **本版不做**：通知模版的发送逻辑、触发条件、渠道配置不变；不新增/删除任何模版；不区分审批人与签署人各自收到不同模版（后续规划）
- **后续规划**：根据接收方角色动态下发不同文案（审批人收"待审批"、签署人收"待签署"），需接入角色判断逻辑

### 功能模块

| 模块名 | 类型 | 说明 |
| --- | --- | --- |
| 通知模版内容管理 | 调整 | 仅修改模版文案字段，不涉及模版结构和触发逻辑 |

### 页面结构

```
无前端页面变更
└── 变更范围限于后台通知模版数据（内容字段）
```

### 核心流程

- **变更范围**：以 Excel "本次调整标注" tab 为唯一变更依据，按行号（数据表行号）逐条定位并修改对应模版的文案字段
- **覆盖渠道**：短信（72条）/ 邮件（83条）/ Lark（66条）/ WhatsApp（49条）/ 站内信（39条）
- **覆盖语言**：英语、西班牙语、日语、简体中文、繁体中文，及葡萄牙语、泰语、印尼语、越南语、马来语、菲律宾语、德语、法语、俄语、意大利语、韩语、阿拉伯语（共17种）
- **覆盖触发场景**：签署任务推送/过期前1天提醒/签署人转交、签署任务过期、信封完成、催签、信封取消、拒签、签署人签署完成（共7类）
- **主流程**：研发按 Excel 行号定位模版 → 对照"调整前/调整后"列替换文案 → 不改动任何变量占位符（如 `{deadline}`、`{envelopeSignUrl}`）
- **验收方式**：QA 对照 Excel 逐条核查调整后文案，确认全部 309 条无漏改、无误改、变量占位符完整保留

### 关键设计决策

| 决策点 | 方案 A | 方案 B | 推荐 & 理由 |
| --- | --- | --- | --- |
| 文案替换粒度 | 统一用"待处理"替换全部"待签署" | 按角色动态下发不同文案 | 推荐 A：本版无角色判断逻辑，B 需额外开发，优先解决体验问题，后续再精细化 |
| 变更依据 | 以 Excel "本次调整标注" tab 为准 | PM 逐条口述变更 | 推荐 A：Excel 已逐条标注调整前/后，减少沟通成本和遗漏风险 |

### 风险提示

- P1：17 种语言的文案均有调整，需每种语言都有对应 QA 或母语者校对，防止非中/英文语言漏测
- P2：变量占位符（如 `{deadline}`、`{envelopeSignUrl}`、`{senderName}` 等）在替换过程中不得被修改或删除，否则通知内容将出现乱码或链接失效
- P3：Lark/WhatsApp 渠道的模版可能需要平台审核（尤其 WhatsApp 模版变更须重新提审），需提前确认是否有审核周期影响排期
