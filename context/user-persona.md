## 画像全景

```
eSignGlobal 外部用户
├── 发起方（Sender）         — 创建并发送信封的人
├── 签署方（Signer）         — 接收并完成签署的人
├── 企业管理员（Admin）      — 配置和管理工作空间的人
├── 抄送方（CC）             — 接收通知但无需操作的人
```

## 画像一：发起方（Sender）

**代表角色**：HR 专员、销售经理等业务人员，负责发起合同签署流程

### 背景
- **技术水平**：因客户而异，通常为普通办公软件用户，无需编程能力
- **使用频率**：因业务场景而异；HR 批量场景为高频，销售场景为中频
- **与产品关系**：核心使用者，是签署流程的发起人和配置责任人

### 核心目标
1. 按业务场景快速完成控件配置并发出信封
2. 确认签署方已收到通知并完成签署
3. 归档和管理已完成的合同文件

### 主要痛点
- **控件配置繁琐**：每次发起都需要手动设置签署区、填写控件的位置和属性，重复且耗时
- **多场景切换成本高**：HR 合同、销售合同配置方式不同，缺乏针对业务场景的快速发起路径

### 典型使用场景
1. **HR 批量发劳动合同**：入职季批量生成合同 → 配置签署控件 → 批量发送 → 追踪签署进度
2. **销售发商务协议**：谈判结束 → 上传协议 → 配置双方签署区 → 发送给客户 → 归档已签版本

### 对产品设计的影响
- 模版能力（信封模版/文件模版）是减少重复配置的核心抓手
- 批量发送、SmartForm 等功能直接服务高频发起场景
- 控件配置流程需最大程度简化，降低每次发起的操作负担

## 画像二：签署方（Signer）

**代表角色**：企业员工（如新入职员工签劳动合同）或外部合作方（如客户、供应商签商务合同）

### 背景
- **技术水平**：因人而异，从完全不懂数字工具到熟练用户均有
- **使用频率**：低至中频，通常为被动触发（收到通知才操作）
- **与产品关系**：被动参与者，无需注册账号即可完成签署（匿名签模式）

### 核心目标
1. 顺利完成签署，不出现失败或中断
2. 快速理解需要在哪里签、填什么内容
3. 确认自己的签署具有法律效力

### 主要痛点
- **签署失败**：因网络、浏览器兼容性或操作不当导致签署流程中断
- **身份认证卡点**：短信验证码、政府 ID 认证等身份核验步骤容易失败或流程不清晰

### 典型使用场景
1. **指定位置签**：收到邮件通知 → 点击链接 → 在发起方预设的位置签名/填写 → 提交完成
2. **自由签**：收到通知 → 点击链接 → 自行拖入签名控件到合适位置 → 提交完成
3. **匿名签转登录签**：通过匿名链接打开 → 发现需要使用已有图章 → 点击登录 → 切换为登录签继续完成

### 关键产品概念（影响设计的术语）

| 签署类型 | 说明 |
|----------|------|
| **指定位置签** | 发起人预设控件位置和类型，签署人只能在指定位置签署和填写 |
| **自由签** | 发起人不预设位置，签署人可拖入签名控件（不可拖填写控件） |
| **登录签** | 签署人登录账号后签署，可访问账号资源（图章、签名等） |
| **匿名签** | 签署人通过匿名链接签署，无需登录，无法访问账号资源；可点击登录按钮转为登录签 |

### 对产品设计的影响
- 签署流程必须极简，步骤越少越好，首次使用者无需引导即可完成
- 身份认证（iAM Smart、Singpass、政府 ID）的失败率直接影响签署完成率，是核心质量指标
- 匿名签/登录签的切换路径需清晰，避免用户在两种模式间感到困惑

## 画像三：企业管理员（Admin）

**代表角色**：IT 负责人或行政负责人，负责为企业配置和管理 eSignGlobal 工作空间

### 背景
- **技术水平**：中等至较高，能理解系统配置概念；具体技术水平因客户而异（IT 专职 or 行政兼职）
- **使用频率**：低频，主要集中在初始配置阶段和权限调整时
- **与产品关系**：工作空间的管理者，负责团队能正常使用产品

### 核心目标
1. 顺利完成工作空间的初始配置（成员、权限、印章等）
2. 维护团队的签署资源和权限体系，确保合规使用
3. 在出现问题时能快速定位和处理

### 主要痛点
- **配置项复杂**：工作空间配置、角色权限、印章管理等选项多，初始上手难度高
- **不清楚配置影响范围**：修改某个配置项时，不知道会对团队现有流程产生什么影响

### 典型使用场景
1. **初始化工作空间**：开通账号 → 邀请成员 → 配置角色权限 → 上传企业印章 → 设置默认签署规则
2. **日常维护**：新员工入职加入工作空间 → 离职员工权限回收 → 更换印章资源

### 对产品设计的影响
- 配置类功能需提供清晰的影响范围说明，降低配置错误风险
- 引导式初始化流程（Onboarding）是管理员成功上手的关键
- 多工作空间管理（Advanced 版本）需要清晰的切换和隔离机制

## 画像四：抄送方（CC）

**代表角色**：需要知悉合同进展但无需操作的相关方，如项目经理、财务、法务等

### 背景
- **技术水平**：不限，通常只需要查看文件
- **使用频率**：极低频，被动接收通知
- **与产品关系**：旁观者，只接收通知和查阅最终文件，无需签署操作

### 核心目标
1. 及时获知合同签署进展和结果
2. 在需要时能查阅已完成的合同文件

### 主要痛点
- 通知文案不清晰时，不确定自己是否需要操作（与签署方通知混淆）

### 典型使用场景
1. 收到「信封已完成」通知 → 点击链接查看最终签署文件 → 下载归档

### 对产品设计的影响
- 发送给抄送方的通知文案必须清晰区分「无需操作，仅供知悉」，避免与签署方通知混淆
- 查看已完成文件的路径需简单，无需登录账号

## 画像五：开发者（Developer）

**代表角色**：负责将 eSignGlobal 集成到企业自有系统（ERP/HR/CRM 等）的技术人员

### 背景
- **技术水平**：高，熟悉 REST API、OAuth、Webhook 等技术概念
- **使用频率**：集成阶段高频，上线后转为低频维护
- **与产品关系**：技术集成者，通过 API 将签署能力嵌入客户自有业务系统

### 核心目标
1. 快速理解 API 能力边界，评估能否满足集成需求
2. 顺利完成集成开发，上线后稳定运行
3. 遇到问题时能快速自助解决，减少人工沟通

### 主要痛点
- **文档不清晰**：API 文档描述模糊或缺失，需要人工咨询才能理解接口行为
- **自助解决困难**：沙箱环境、错误码说明不够完善，遇到问题必须联系支持

### 典型使用场景
1. **评估集成可行性**：查阅 API 文档 → 确认能力边界 → 向团队汇报集成方案
2. **开发集成**：申请 API Key → 在沙箱中调试 → 联调签署流程 → 上线生产环境
3. **上线后维护**：监控 Webhook 通知 → 处理异常状态的信封 → 随版本更新调整集成逻辑

### 对产品设计的影响
- API 文档质量直接影响开发者体验，是留存开发者客户的核心资产
- 沙箱环境、错误码、Webhook 可靠性是开发者最关注的质量指标
- 开发者自助能力越强，客服支持成本越低

## 用户优先级参考

| 用户类型 | 优先级 | 设计权重 |
|----------|--------|----------|
| 发起方 | P0 | 核心服务对象，签署流程体验的主要负责人，所有发起类功能围绕其工作流设计 |
| 签署方 | P0 | 签署完成率直接影响产品核心价值，签署体验决定客户对产品的第一印象 |
| 企业管理员 | P1 | 影响客户成功上手和续约，配置类功能围绕其需求设计 |
| 开发者 | P1 | API 集成客户的留存和扩展关键，文档和开发者体验需持续投入 |
| 抄送方 | P2 | 被动参与者，设计保证通知清晰、查看便捷即可，不为其定制操作界面 |
