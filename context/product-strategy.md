# 产品策略

> 写 PRD 时遇到「要不要做」「这期做还是下期做」「边界在哪」的判断，以此文件为准。

---

## 1 核心迭代策略原则

### 原则一：Roadmap 优先，客户需求是例外而非常态

**实操含义**：
- 季度 Roadmap 是主干，每双周迭代按 Roadmap 排定内容推进，不因单个客户需求随意调整计划
- 收到客户需求时，第一步不是评估「要不要做」，而是评估「是否满足 P0 插队条件」
- 不满足则进入下季度 Roadmap 排期流程，不占用当期资源

### 原则二：标准化产品，不做客户定制

**实操含义**：
- 所有需求必须对产品整体有价值，不为单一客户的特殊场景开发专属功能
- 客户反馈是需求来源，但最终方案必须是可复用的标准能力
- 如果一个需求只有一个客户提出且无法抽象为通用场景，直接拒绝；多客户同诉求才考虑排入 Roadmap

### 原则三：兼容性优先于创新速度

**实操含义**：
- 新功能上线不得破坏老客户的现有流程，发起流程、匿名签署等核心链路尤其需要向下兼容
- 所有涉及核心链路的需求，必须在写 PRD 前完成兼容性影响评估（见需求前检查清单），不能在评审阶段才发现兼容问题

---

## 2 需求优先级矩阵

### 分级定义

| 优先级 | 触发条件 | 典型场景 | 决策结果 |
|--------|----------|----------|----------|
| **P0（立即处理）** | 卡单 / 阻断客户业务流程运转 | 客户无法完成签署、API 故障导致业务中断 | 插入当期迭代，Roadmap 顺延 |
| **P1（计划内）** | 季度 Roadmap 已排定需求 | 季度规划会议确认的功能 | 按双周迭代计划推进 |
| **P2（排期评估）** | 客户需求，不卡单但影响续约/增购 | 大客户强烈诉求、多客户重复反馈 | 评估后排入下季度 Roadmap |
| **P3（不做）** | 单客户定制、与产品方向不符 | 只有一个客户提出的特殊场景 | 明确拒绝，不进入排期 |

### 客户需求插队判断流程

```
客户提出需求
    │
    ▼
是否卡单 / 阻断业务流程？
    │
    ├─ 是 → P0，插入当期迭代
    │
    └─ 否 → 评估客户体量（合同金额 + 影响客户数）
                │
                ├─ 体量大 / 多客户同诉求 → P2，排入下季度 Roadmap
                │
                └─ 单客户 / 体量小 → 拒绝或积压观察
```

---

## 3 边界约束（明确不做）

| 不做的事 | 原因 |
|----------|------|
| 客户定制开发 | 任何功能必须对产品整体有价值，不为单一客户开发专属逻辑 |
| 在当期迭代随意插入非 P0 需求 | Roadmap 稳定性是团队执行效率的保障，非 P0 插入破坏计划节奏 |

## 4 写需求前的检查清单

> 动笔写 PRD 之前逐项过一遍。任何一项有影响，必须在 PRD 中显式说明处理方案。

### 通用检查项（所有需求必查）

- [ ] **数据中心影响**：该功能是否因数据中心不同（香港/新加坡/法兰克福）而有差异行为？数据是否会跨区域流动？
- [ ] **合规影响**：是否涉及电子签名法律效力、GDPR、eIDAS、ESIGN/UETA 等合规要求？AES/QES 签名等级是否受影响？
- [ ] **匿名签署流程影响**：该功能是否影响签署方无需登录/注册即可完成签署的流程？
- [ ] **老客户兼容**：已在使用该功能相关模块的老客户，升级后行为是否保持一致？是否需要数据迁移或灰度策略？
- [ ] **多语言/国际化影响**：该功能涉及的文案、通知、提示是否覆盖全部支持语言（当前 17 种）？是否有语言相关的展示差异（如 RTL 布局、字符长度）？

### 发起流程专属检查项（涉及「发起」模块时必查）

- [ ] **草稿**：对草稿状态的信封是否有影响？
- [ ] **批量发送**：批量发起场景下的行为是否与单个发起一致？
- [ ] **单个发送**：单份发起的正常流程是否受影响？
- [ ] **API 发起**：通过 API 发起信封的场景是否同步覆盖？
- [ ] **编辑（修改已发起信封）**：是否影响已发起但未完成的信封的编辑逻辑？
- [ ] **发送者视图（Sender View）**：发起方看到的视图是否有变化？
- [ ] **复制信封**：复制已有信封重新发起的场景是否受影响？
